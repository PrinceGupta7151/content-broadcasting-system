# Content Broadcasting System - Frontend Notes

## Project Structure

The frontend is organized for clarity and role separation:
- `src/components/`: Reusable UI building blocks and shared widgets
  - `common/`: Navbar, Sidebar, Loading, Toast
  - `auth/`: Login-specific components
  - `teacher/`, `principal/`, `public/`: role-specific UI pieces
- `src/pages/`: Route-level page components
  - `auth/`: Login page
  - `teacher/`: Dashboard, Upload, MyContent
  - `principal/`: Dashboard, PendingApprovals, AllContent
  - `public/`: Live broadcast viewer
- `src/services/`: API abstraction layer using Axios
  - `api.js`: Axios instance with interceptors for auth and error handling
  - `auth.service.js`, `content.service.js`, `approval.service.js`
- `src/hooks/`: Custom hooks for app logic
  - `useAuth.js`, `useApi.js`, `useToast.js`
- `src/context/`: Context providers
  - `AuthContext.jsx`, `ToastContext.jsx`
- `src/utils/`: Shared utilities
  - `constants.js`, `validators.js`, `formatters.js`, `storage.js`
- `App.jsx`: Root component with route definitions and protected route setup

## Authentication Flow

1. User opens `/login` and submits email/password.
2. The login form calls `auth.service.login()`.
3. On success:
   - JWT token is stored in `localStorage`.
   - User profile and role are stored in `AuthContext`.
   - The app redirects the user based on role.
4. `ProtectedRoute` checks auth state and optionally verifies role before allowing access.
5. The Axios instance in `api.js` attaches the stored token to all outgoing requests.
6. If the backend returns `401`, the interceptor triggers logout and redirects to `/login`.

## Role-Based Routing

- Teacher routes are grouped under `/teacher` and include:
  - `/teacher/dashboard`
  - `/teacher/upload`
  - `/teacher/content`
- Principal routes are grouped under `/principal` and include:
  - `/principal/dashboard`
  - `/principal/pending`
  - `/principal/content`
- Public routes require no auth and include:
  - `/login`
  - `/live/:teacherId`

Role checks are enforced by `ProtectedRoute`, which ensures the current user is authenticated and has the correct role before rendering the route.

## API Integration Approach

- All backend communication is centralized in service modules.
- `api.js` exports an Axios instance with:
  - Authorization header injection
  - Response error handling
- Service files expose domain-specific functions:
  - `auth.service.js` handles login/logout and token validation
  - `content.service.js` handles content fetch, upload, and details
  - `approval.service.js` handles pending approvals and status updates
- Components call service functions, not Axios directly.
- This keeps API logic separate from UI and makes switching the backend or adding mocks easier.
- Errors are surfaced through `useApi` and shown with Toast notifications.

## State Management Approach

- `AuthContext` manages global authentication state and the current user.
- `ToastContext` manages notification state for success/error alerts.
- Local component state handles UI details such as form values, loading flags, and modal visibility.
- `React Hook Form` manages form state and validation.
- Custom hooks encapsulate reusable behavior:
  - `useAuth()` exposes auth state and actions.
  - `useToast()` exposes toast dispatch functions.
  - `useApi()` wraps API calls with loading/error handling.
- This approach avoids introducing Redux for this app size while keeping global state simple and predictable.

## Assumptions

- The backend exposes a REST API reachable via `process.env.REACT_APP_API_URL`.
- Authentication returns a JWT token.
- The token is stored in `localStorage` and used for future requests.
- Role information is part of the user object.
- File uploads are sent as `multipart/form-data`.
- Content timestamps use ISO 8601 format.
- Public live broadcasts are served via a route that does not require auth.
- User roles are limited to at least `teacher` and `principal`.

## Setup Instructions

1. Install dependencies:
   npm install

2. Create .env file:
   REACT_APP_API_URL=http://localhost:5000/api 

3. Start development server:
   npm run dev

4. Build for production:
   npm run build

## API Endpoints Required

### Authentication
- POST /auth/login
- POST /auth/logout
- GET /auth/verify

### Content
- GET /content/teacher (teacher's content)
- GET /content/all (principal's all content)
- POST /content/upload (upload new content)
- GET /content/:id (get specific content)
- GET /content/live/:teacherId (public live broadcast)

### Approval
- GET /approval/pending (pending approvals)
- POST /approval/:id/approve (approve content)
- POST /approval/:id/reject (reject with reason)
- GET /approval/stats (dashboard statistics)
